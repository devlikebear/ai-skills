# Tidy First Rules (Kent Beck) - Source Analyzer Summary

Use this file when producing `TIDY-*` issues and work orders.
Apply rules as concise, actionable guidance in `refactor-*.md`.

## Core Rules

1. **구조 변경과 동작 변경을 분리한다**
- 먼저 구조를 정리하고, 그 다음 동작을 바꾼다.
- 한 커밋/한 WO에서 구조+동작을 동시에 크게 바꾸지 않는다.

2. **작고 안전한 단계로 진행한다**
- 큰 재작성보다 작은 추출/이동/이름변경을 우선한다.
- 각 단계는 되돌리기 쉬워야 한다.

3. **동작 보존을 먼저 보장한다**
- 구조 정리 단계에서는 관측 가능한 동작 변화가 없어야 한다.
- 테스트가 없다면 최소 회귀 검증 루트를 먼저 만든다.

4. **국소 정리(local tidy)를 우선한다**
- 지금 수정 중인 코드 주변을 먼저 정리한다.
- 전체 코드베이스 대정리는 별도 WO로 분리한다.

5. **의도를 드러내는 이름을 사용한다**
- 모호한 이름을 정리해 읽기 비용을 줄인다.
- 의미 없는 축약/단문 변수명을 제거한다.

6. **중복을 통제 가능한 단위로 모은다**
- 반복되는 분기/매핑/검증 로직은 공통화한다.
- 공통화는 책임 경계를 넘지 않는 선에서 수행한다.

## `TIDY-*` 작성 규칙

- 모든 `TIDY-*` 항목에 다음을 명시한다.
  - 적용 룰 이름
  - 구조 변경 단계
  - 동작 변경 단계(필요 시)
  - 테스트 기준(단위/통합/회귀)
- 구조 변경만 수행하는 WO와 동작 변경 WO를 분리할 수 있으면 분리한다.
